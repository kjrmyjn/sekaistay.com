## 2. Guesty 分析からの抽出（詳細は補足参照）

Guesty の HP 設計が際立つ 3 点:
1. **段階的ペルソナ分化の明快性** — 1物件 / 中規模 / 大規模 を価格3層で明示
2. **数字証拠の具体性と多様性** — 500K+ 物件 + 国別実績（Chile 10x 等）
3. **効果ファーストのフレーミング** — 「150% revenue growth」が見出し、機能名は後

## 3. SEKAI STAY 文脈への翻訳（仮説 5項目）

### 3.1 ペルソナ 3層化

**現状**: PainPoints セクションで「運用中／新規」の 2層
**改善**: 物件規模軸で 3層に細分化

| 層 | ペルソナ | 想定 CTA |
|---|---|---|
| L1 | 1物件オーナー（個人投資家・副業）| まずシミュレーション（30秒・軽い入口）|
| L2 | 複数物件オペレーター（3-10物件）| 無料診断レポート（中重さ・現行 /audit）|
| L3 | 大規模・新規開業（10物件以上・ファンド）| 個別相談予約（重い入口・timerex 直結）|

**Why**: 「運用中／新規」軸は心理状態を分けるが、**意思決定の重さ**を分けない。規模軸の方が「自分の話」と感じやすく、適切な CTA に誘導できる。

**触る場所**:
- `components/home/PainPoints.tsx` の personas[] を 3 件に
- `content/home/copy.ts` の `PAIN_POINTS.personas` 更新
- 各ペルソナカードに**そのペルソナ向けの CTA**を埋め込み（現状は共通 CTA）

### 3.2 Hero に「数字3つ + カテゴリリーダー宣言」を統合

**現状**: Hero はヘッドライン + 段落コピー + CTA + 画像で構成。数字は直下の AuthorityBar に分離。
**改善**: Hero 内に**数字3つ + カテゴリリーダー宣言**を融合し、1秒で信頼を作る

**コピー案**:
```
[Hero ヘッドライン]
成果で選ばれる、日本の民泊運営代行。

[数字3つ - Hero 内ライン]
管理物件 ★4.8 / 平均稼働率 +30% / 国土交通大臣 第F05780号

[CTA]
無料で物件診断を受ける
```

AuthorityBar セクションは**削除**または**Credentials 表示専用**に格下げ。

**Why**: Guesty が「The Leading AI property management software」+ 500K/99.99% で1秒で勝負しているのと同型。現状の Hero は数字が散ってて 1秒目の主張が弱い。

**触る場所**:
- `components/home/Hero.tsx` のレイアウト改修
- `components/home/AuthorityBar.tsx` 削除 or 内容変更
- `content/home/copy.ts` の `HERO` / `AUTHORITY` 構造更新

### 3.3 機能カテゴリ 3バケット化

**現状**: /services は 9 サービスをフラットに並列表示
**改善**: 3カテゴリで束ねる（Guesty の "Guests & Reservations / Distribution & Operations / Business & Financials" 型）

| カテゴリ | 含むサービス |
|---|---|
| **集客・予約** | OTA最適化 / 多言語ゲスト対応 / 価格設計（レベニューマネジメント）|
| **運営・現場** | 清掃手配 / トラブル対応 / 緊急対応 / リネン・消耗品 |
| **数字・透明性** | オーナーダッシュボード / 月次レポート / 売上分析 / 改善提案 |

**Why**: 9個フラットだとどれが重要か分からない。3つに束ねると「全部やってる」と一覧で伝わる。

**触る場所**:
- `app/services/page.tsx` 構造改修
- `content/services/copy.ts`（あれば）

### 3.4 効果ファーストのコピー転換

**現状**: 事例カードの見出しは物件名（例: 「野尻湖エリアの貸別荘」）、効果は本文内に
**改善**: 効果が見出し、物件情報は subline

**Before**:
```
[Case 01]
野尻湖エリアの貸別荘
リゾート物件 / 稼働率 58% → 82% / 月商 +57%
```

**After**:
```
[Case 01]
月商 +57% / 稼働率 1.4倍
野尻湖エリアの貸別荘・リゾート物件
```

**Why**: Guesty Customers が「150% revenue growth」を最前面、機能説明は後にしているのと同じ思考。決裁者は**実績の数字**を見たい。

**触る場所**:
- `components/home/Results.tsx` の事例カードレイアウト
- `app/case-studies/page.tsx`（あれば）も同様に
- `content/home/copy.ts` の `RESULTS.cases[]` 構造（headline と subline 追加）

### 3.5 CTA を 3段ファンネル化

**現状**: 主要 CTA は「無料で物件診断を受ける」(/audit) に一本化。FloatingCTA も同様。
**改善**: コミット度の異なる 3 つの CTA で離脱層を拾う

| 段 | コピー | 遷移先 | 想定ユーザー |
|---|---|---|---|
| **軽** | 30秒で年間損失額を試算 | /services#pricing（計算機）| 「自分の物件で試したい」だけの低コミット |
| **中** | 無料物件診断レポート | /audit | レポートで詳細を知りたい |
| **重** | 個別相談予約 | timerex直結 | 即話したい |

**Where**:
- Hero: 軽 + 中（PT・SP 両方）
- MidCTA: 中だけ（現状維持）
- NavCards: 軽 + 中 + 重 の 3 カードに変えてもいい
- FinalCTA: 重 中心、軽もテキストリンクで

**Why**: 現状は「重」しかないので、低コミット層が「どうせ営業されるんだろう」と離脱。3段に分けると、自分のコミット度に合った入口が見つかる。

**触る場所**:
- `components/home/Hero.tsx`（軽 CTA 追加）
- `components/home/NavCards.tsx` または FinalCTA（重 CTA 追加）
- `content/home/copy.ts` の CTA_LABELS 拡張

## 5. スコープ外（やらない）

| 項目 | やらない理由 |
|---|---|
| AI / 機械学習訴求 | 嘘になる（運営は人手中心） |
| Self-serve 料金プラン | サービス業態に合わない |
| 海外実績の地理多様性 | 国内限定 |
| Series 資金調達アピール | 文化的に逆効果 |
| Enterprise tier の独立化 | 中規模特化なので不要 |

## 6. テスト戦略

このフェーズはコピー/構成変更が中心で機能変更ではない。

- ローカル: `npm run dev` で目視確認
- ビルド: `npm run build` 通過
- preview: Vercel preview で実機確認（特に**モバイル**で数字が崩れないか）
- 本番後: GA / Clarity で /audit submit 完了率を 1週間モニタ（Phase A 比較）

## 7. オープン課題

実装時に確定する:
- **3.2 Hero の数字3つの選定** — 「管理物件数」「★4.8」「稼働率 +30%」が候補だが、最新の実数値を引用する必要あり（運営から事実確認）
- **3.5 軽 CTA のコピー** — 「30秒で年間損失額を試算」は仮。A/B テストの余地あり
- **3.1 ペルソナ 3層の境界値** — 「3-10物件」と「10物件以上」の境界は要相談
- **3.4 効果ファーストのフレーズ** — 「+57%」を見出しにすると数字疲れの懸念。複数パターン用意して選定
